Authentication, Authorization, Refresh Token, 401 and 403 Story Guide

A print-ready interview and explanation guide that walks through login, token usage, authorization decisions, refresh flow, and failure scenarios like a story instead of a disconnected checklist.

Authentication

Proves who the user is. Usually ends with an access token and a refresh token.

Authorization

Checks what the authenticated user is allowed to do after identity is already known.

Refresh Flow

Recovers a short-lived access token without forcing the user to log in again.

1. The Story in One Smooth Narrative

Think of the full system as a secure office building. The user first goes to reception and proves identity using credentials like email and password. Once verified, the system gives the user a short-lived badge, which is the access token. That badge is shown every time the user tries to enter a protected room such as dashboard, orders, or admin APIs.

But the badge expires quickly for safety. So the system also keeps a stronger, more protected credential, which is the refresh token. The refresh token is usually hidden in an HttpOnly cookie so browser JavaScript cannot read it easily. When the access token expires, the client quietly presents the refresh token to the backend, gets a new badge, and continues the user journey with minimal friction.

After authentication, authorization takes over. The backend looks at the user identity, roles, scopes, tenant, and business rules to decide whether the user may perform the requested action. If identity is missing or invalid, the server returns 401 Unauthorized. If identity is valid but permission is insufficient, the server returns 403 Forbidden.

3. Authentication: How It Works

Authentication

Authentication begins when the user submits credentials or an external identity provider response. The backend verifies those credentials against the source of truth. If they are correct, the backend creates an identity session and issues tokens.

  1. User enters credentials.
  2. Backend validates password, MFA, or OAuth result.
  3. Backend issues an access token with a short expiry.
  4. Backend issues a refresh token with stronger protection.
  5. Frontend stores the access token carefully and starts normal API usage.
Interview phrase: authentication answers who are you?

4. Authorization: How It Works

Authorization

Authorization only starts after identity is known. The backend evaluates whether the identified user may perform a specific action on a specific resource. Roles are only one part of this; mature systems also check scopes, tenant ownership, account status, and resource-level rules.

  1. Backend reads and verifies the access token.
  2. Backend resolves user identity and claims.
  3. Backend loads policy context such as role, tenant, or object ownership.
  4. Backend decides allow or deny.
  5. If denied because of permissions, return 403.
Interview phrase: authorization answers what are you allowed to do?

5. Access Token vs Refresh Token

Token Model
  • Access token: short-lived, sent on API calls, higher exposure, should expire quickly.
  • Refresh token: long-lived, used only to obtain a new access token, should be much better protected.
  • Common design: keep access token in memory and refresh token in an HttpOnly cookie.
  • Security idea: the credential that travels more often should be less powerful and shorter-lived.

6. Refresh Token Story

Refresh Handling

The refresh token flow exists because access tokens should expire frequently. A mature client should not immediately panic on a single 401. It should first ask: “Did the access token simply expire while the session is still valid?”

  1. Request fails with 401.
  2. Client checks whether a refresh flow is already in progress.
  3. If not, start exactly one refresh request.
  4. If refresh succeeds, update access token and retry queued requests.
  5. If refresh fails, clear session and redirect to login.

7. 401 vs 403: The Difference

401 Unauthorized

This means the server could not accept the current authentication state. Either there was no token, the token was expired, the signature was invalid, the issuer was wrong, or the token was revoked.

  • No token sent
  • Access token expired
  • Malformed token
  • Invalid signature
  • Revoked session or blocked token
403 Forbidden

This means the server does know who the user is, but still refuses the action. The token may be valid, but the policy decision is deny.

  • User lacks admin role
  • User is outside allowed tenant
  • User has read scope but not write scope
  • Account is authenticated but suspended from this operation
Smooth interview line: 401 means “I cannot trust your identity right now,” while 403 means “I trust who you are, but you still cannot do this.”

8. Real Edge Cases Like a Story

Scenario 1

Five API calls fail at the same time with 401

This is the classic concurrency problem. If all five requests independently trigger refresh, the client may create a refresh storm. In a rotation-based system, that can even invalidate tokens unexpectedly.

The safer story is single-flight refresh: one request performs refresh, the others wait, and after the new access token arrives, all queued requests retry using the same fresh token.

Scenario 2

Access token is expired, but refresh token is still valid

This is the normal recovery path. The user should not be sent back to login immediately. The client should refresh quietly, update the access token, and continue without visible interruption.

Scenario 3

Refresh token is expired or revoked

Now the session is really over. The client should stop retrying protected requests, clear local auth state, and redirect the user to login. Continuing to retry in a loop is both noisy and broken.

Scenario 4

User is authenticated but role changed on the server

The access token may still contain old claims until it expires. That is why access tokens should be short-lived, and why critical authorization decisions must always be enforced on the backend rather than trusting the frontend alone.

Scenario 5

Valid access token, but user tries to open an admin route without permission

Here the system should not refresh the token just because the request failed. This is a permission failure, not an authentication failure. The backend should return 403, and the frontend should show an access denied state.

Scenario 6

Page reload clears in-memory access token

If the app stores access tokens only in memory, a full reload removes them. If the refresh cookie is still valid, the app can silently restore the session on startup by calling refresh once.

Scenario 7

Refresh token reuse detected

In rotated refresh-token systems, reuse of an old refresh token suggests theft or replay. The backend should revoke the token family and force re-authentication because the session can no longer be trusted.

9. Frontend Behavior Guidelines

  • Attach the access token to protected API requests only.
  • Never attach Bearer tokens to login, refresh, or logout endpoints unless intentionally designed.
  • On 401, attempt one coordinated refresh flow.
  • On 403, do not refresh by default; show denied UI or redirect to a forbidden page.
  • Queue concurrent failed requests behind a single refresh operation.
  • On refresh failure, clear local session state immediately.

10. Backend Behavior Guidelines

  • Validate token signature, issuer, audience, and expiry on every protected request.
  • Use short-lived access tokens to reduce stale authorization windows.
  • Protect refresh tokens with cookies, rotation, revocation, and reuse detection where possible.
  • Return 401 when authentication is invalid.
  • Return 403 when authentication is valid but permission is denied.
  • Do not trust frontend role checks as real security.

11. Interview Closing Answer

If I had to explain the whole model smoothly, I would say this: authentication verifies the user and issues a short-lived access token plus a protected refresh token. The access token is used for normal API requests, while the refresh token quietly restores the session when the access token expires. Authorization happens after identity is established, and it decides whether the authenticated user may perform a specific action.

When things go wrong, the status code tells the story. A 401 means the system cannot accept the current authentication state, so refresh may be needed. A 403 means the identity is known, but the action is still not allowed, so refreshing the token usually does not help. The best systems handle all of this quietly, safely, and without turning token expiry into a bad user experience.

Use this as a speaking script: explain the happy path first, then explain expiry recovery, then explain why 401 and 403 mean very different things. That order sounds clear and senior in interviews.
